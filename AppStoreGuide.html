App Store Submission Guide

Everything you need to build, configure, upload, and publish FarmerChat on the Apple App Store — from first build to live release.

iOS Xcode 15+ Bundle ID: org.digitalgreen.farmer.chatbot Team: 49VAPTB77A
ℹ️

iOS vs Android: Android distributes apps as .apk files. iOS distributes as .ipa files (iOS App Archive). Xcode handles creating the IPA — you just archive and upload.

You need an active Apple Developer Program membership ($99/year USD). Your existing Team ID is 49VAPTB77A.

  • Enroll at developer.apple.com/programs if not already enrolled.
  • The account email becomes your App Store Connect login.
  • Your account must have App Manager or Admin role to submit apps.

Certificates & Signing

Your project uses Automatic Code Signing — Xcode manages certificates for you. To verify:

  1. Open FarmerChat.xcodeproj in Xcode.
  2. Click the FarmerChat project in the navigator → select the FarmerChat target → Signing & Capabilities tab.

  4. Confirm Team is set to your Digital Green team (49VAPTB77A).

Xcode Project Configuration for Production

Version Numbers

In Xcode → FarmerChat target → General tab:

FieldCurrentDescription
Version (Marketing)1.0Shown to users on App Store. Increment for user-facing changes.
Build (Project Version)1Must be unique and increasing with every upload to Apple.
🚫

Critical rule: The Build number must always be higher than the last uploaded build. Apple rejects duplicate build numbers. Even for a hotfix on the same version, bump Build from 1 → 2.

Set Release Build Configuration

  1. Click the scheme selector (top bar in Xcode) → Edit Scheme…
  2. Select Archive in the left panel.
  3. Set Build Configuration to Release.

Privacy Usage Descriptions (Info.plist)

Apple requires a plain-English reason for every privacy API your app accesses. Missing strings cause automatic rejection.

Plist KeyRequired If
NSCameraUsageDescriptionApp accesses the camera
NSMicrophoneUsageDescriptionApp records audio
NSLocationWhenInUseUsageDescriptionApp uses location services
NSPhotoLibraryUsageDescriptionApp reads from photo library

Verify Entitlements

Your app's entitlements file is at FarmerChat/FarmerChat.entitlements. Open it and ensure only capabilities you actually use are listed. Unused entitlements trigger App Review rejections.

Building a Production IPA

An IPA is the iOS equivalent of an Android APK. Here is the full build process:

Step 1 — Select the Right Destination

In Xcode's scheme selector, set the destination to Any iOS Device (arm64) — NOT a simulator. Simulator builds cannot be submitted to Apple.

Xcode ToolbarFarmerChat  ▶  Any iOS Device (arm64)

Step 2 — Archive the App

MenuProduct  →  Archive

Xcode compiles a Release build and adds it to the Organizer. This takes 1–5 minutes. Wait for the spinner to complete.

Step 3 — Open Organizer

MenuWindow  →  Organizer

You'll see your new archive listed with today's date and version number.

Step 4 — Distribute the App

  1. Select your archive in Organizer → click Distribute App.
  2. Choose App Store Connect → Next.
  3. Choose Upload → Next.
  4. Leave all checkboxes at defaults (Strip Swift Symbols, Upload Symbols) → Next.
  5. Select Automatically manage signing → Next.
  6. Review the summary → click Upload.
💡

Need the IPA on disk? In step 3 above, choose Export instead of Upload. This saves the .ipa file locally. You can then drag it into the Transporter app to upload it later.

5

Uploading to App Store Connect

After Xcode uploads, Apple processes the build. This takes 10–30 minutes.

Upload Complete
Xcode reports success. The build appears in App Store Connect → TestFlight tab.
Processing
Apple runs automated checks on the binary. Status shows "Processing" (10–30 min).
Missing Compliance (if shown)
Apple asks about encryption. For apps using only HTTPS, answer: No encryption. Or add ITSAppUsesNonExemptEncryption = false to Info.plist to skip this.
Ready to Submit
Build can now be attached to your App Store version for review.

Upload via Transporter (Alternative)

  1. Open Transporter from the Mac App Store.
  2. Sign in with your Apple Developer Apple ID.
  3. Drag and drop your exported .ipa file into the window.
  4. Click Deliver.

Submitting for Apple Review

Attach the Build

  1. App Store Connect → Your App → App Store → 1.0 Prepare for Submission.
  2. Scroll to the Build section → click "+".
  3. Select your uploaded build → click Done.

Fill in Review Information

FieldNotes
Sign-in RequiredYes/No — if Yes, provide a working demo account
Demo Account UsernameA test account Apple reviewers can log into
Demo Account PasswordStable password that won't expire
Notes for ReviewerExplain any non-obvious flows or features
AttachmentOptional video walkthrough — speeds up review
💡

Clear reviewer notes and a working demo account are the single biggest factor in avoiding rejections. Apple reviewers use real devices — don't assume they know your app's context.

Release Options

  • Manually release — you click "Release" after approval. Gives you control over timing.
  • Automatically release — goes live immediately after Apple approves.
  • Scheduled release — goes live at a specific date/time you set.

Submit

Click Add for Review → review the checklist → Submit to App Review.

8

After Approval — Releasing to Users

If you chose Manual Release:

  1. Go to App Store Connect → Your App → App Store.
  2. Click Release This Version.

If you chose Automatic Release: no action needed — Apple publishes it immediately after approval.

ℹ️

After release, the app appears on the App Store within a few hours due to Apple's CDN propagation across regions. It may not be visible immediately in all countries at the exact same time.

9

Updating the App (New Versions)

Every update follows this cycle:

  1. Bump the Build number in Xcode (always required — even for tiny fixes).
  2. Bump the Version number if the change is user-facing (e.g. 1.0 → 1.1).
  3. Archive & Upload — same steps as Section 4.
  4. In App Store Connect → App Store → click "+" next to iOS App (left sidebar) → New Version.
  5. Enter the new version number (must match MARKETING_VERSION in Xcode).
  6. Fill in "What's New in This Version" → attach the build → submit.
💡

You do not need to redo all metadata for an update — only update the "What's New" text and attach the new build. Everything else carries over from the previous version.

Switching to Production Backend

Before every App Store submission, ensure your app points to production servers, not staging or localhost.

Find All Hardcoded Values

Terminal — run from project root# Find all API URLs in Swift files
grep -r "http" FarmerChat/ --include="*.swift" | grep -v "//.*http"

# Find debug flags or environment toggles
grep -r "DEBUG\|isDebug\|isDev\|staging\|localhost" FarmerChat/ --include="*.swift"

Recommended: Build Configuration Flags

Use Xcode's Debug/Release configurations to switch environments automatically — no manual changes needed before each build.

  1. In Xcode → FarmerChat target → Build Settings.
  2. Search for Active Compilation Conditions.
  3. Under Debug: add DEBUG. Under Release: leave empty.
  4. Use #if DEBUG in code to switch values:
Swift — AppConfig.swiftstruct AppConfig {
    static let apiBaseURL: String = {
        #if DEBUG
        return "https://staging-api.farmerchat.digitalgreen.org"
        #else
        return "https://api.farmerchat.digitalgreen.org"
        #endif
    }()
}

Troubleshooting Common Errors

ErrorCauseFix
No accounts with App Store distribution Apple ID not signed in to Xcode Xcode → Settings → Accounts → Add Apple ID
Invalid Binary (upload rejected) Simulator build or wrong configuration Archive with Any iOS Device (arm64), Release config
Missing Compliance prompt Apple asking about encryption use Add ITSAppUsesNonExemptEncryption = false to Info.plist
Minimum OS version mismatch Deployment target too low Xcode → General → Minimum Deployments → set iOS 16.0+
Build stuck in Processing Apple server delay or binary issue Wait 2 hours; if still stuck, delete and re-upload. Check Apple System Status.

Pre-Submission Checklist

Run through this before every App Store submission:

  • Build number incremented (higher than last upload)
  • Version number updated if this is a user-facing change
  • Archive built targeting Any iOS Device (arm64)
  • Archive built with Release build configuration
  • All API URLs pointing to production endpoints
  • Debug logging and verbose output disabled
  • All Info.plist privacy usage descriptions filled in
  • Screenshots uploaded for 6.5" and 5.5" iPhone sizes
  • App description, keywords, and metadata reviewed
  • Demo account credentials provided (if login required)
  • Support URL is live and reachable
  • Age rating questionnaire completed
  • Encryption compliance answered (or ITSAppUsesNonExemptEncryption = false)
  • Entitlements file matches only capabilities actually used
  • Tested on a real physical device (not only Simulator)
